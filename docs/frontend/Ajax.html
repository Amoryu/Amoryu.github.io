//创建路由规则，响应向url为“/”发来的GET请求

 	//设置响应头

    response.send("welcome to Ajax! jiahua sssss");
});

//设置监听端口启动服务
app.listen(8000, () => {
    console.log('服务已启动,8000端口监听中');
})

# Ajax响应超时与网络异常

设置XHR的timeout属性，用于表示发送请求后等待了多少毫秒，如果响应不成功就中断请求(status由pending变为canceled)

xhr.timeout = 1000;		//1秒后响应不成功就中断请求

xhr通过调用ontimeout事件处理程序，timeout属性时间内没有收到响应就会触发ontimeout事件处理程序（此时readystate仍会变为4，所以会触发onreadystatechange事件处理程序，但访问status属性会出错）

xhr.ontimeout = function () {
	alert("网络出现了点问题，请稍后重试");
    console.log(xhr.readyState);
}

# CORS(跨源资源共享)

# Cross-Origin-Resource-Sharing

==W3C标准，跨域的根本解决方案==，支持GET和POST请求，但是缺点是不兼容某些低版本的浏览器

通过XHR进行Ajax通信的一个主要限制是跨源安全策略，默认情况下，XHR对象只能访问与发起请求的页面在同一个域内的资源，这个安全策略可以防止某些恶意行为

不过，浏览器也需要支持合法跨源访问的能力CORS

基本思路：使用自定义的HTTP头部允许浏览器和服务器互相了解

Origin头部，如果服务器决定响应请求，就应该发送Access-Control-Allow-Origin头部，包括相同的源，如果资源是公开的，那么久包含‘ * ’

跨域XHR对象允许访问status和statusText属性，也允许同步请求，但也有如下限制

  • 不能使用setRequestHeader()设置自定义头部
  • 不能发送和接收cookie
  • getAllResponseHeaders()始终返回空字符串

# 预检请求（Preflighted request）

==浏览器==在向服务器发送类似 【==自定义头部、除GET和POST之外的方法、不同请求体内容类型==】 的请求时，会先向服务器发送一个预检请求，这个请求使用OPTIONS方法发送并包含以下头部

  • Origin：与简单请求相同
  • Access-Control-Request-Methods：请求希望使用的方法
  • Access-Control-Request-Headers：想要使用逗号分隔的自定义头部列表（可选）

==服务器==收到这个预检请求后，可以确定是否响应这种类型的请求，通过在响应中加入以下头部与浏览器沟通这些信息

  • Access-Control-Allow-Origin：与简单请求相同(✳为允许跨域)
  • Access-Control-**Allow-**Methods：服务器允许的方法（逗号分隔的列表）
  • Access-Control-Allow-Headers：服务器允许的头部（逗号分隔的列表）
  • Access-Control-Max-Age：缓存预检请求的秒数

# 凭据请求

默认情况下，跨源请求不提供凭据（cookie，HTTP认证和客户端SSL证书），可以通过将withCredentials设置为true表明请求会发送凭据。

如果服务器允许带凭据的请求，则会在响应体中包含以下HTTP头部

  • Access-Control-Allow-Credentials: true :服务器允许带凭据的请求

    如果发送了凭借请求而服务器响应中没有这个头部，则浏览器不会将响应交给JavaScript（responseText是空字符串，status是0，onerror()被调用）

# axios发送Ajax请求

Axios，是一个基于promise网络请求库，作用于node.js和浏览器中，它是 isomorphic 的(即同一套代码可以运行在浏览器和node.js中)。在服务端它使用原生node.js http模块, 而在客户端 (浏览端) 则使用XMLHttpRequests。

# 原理

axios本质上也是对原生XHR的封装，只不过它是Promise的实现版本，符合最新的ES规范。

# 主要特点

  • 从浏览器创建 XMLHttpRequests

  • 从 node.js 创建 http 请求

  • 支持 Promise API

  • 拦截请求和响应

  • 转换请求和响应数据

  • 取消请求

  • 自动转换JSON数据

  • 客户端支持防御XSRF

# 节流和防抖

# 节流throttle

如果事件被频繁出发，节流能减少一段时间内事件的触发频率，因此，节流是有选择性地执行一部分事件

# 应用场景

  1. 鼠标连续不断地触发某事件，只在单位时间内触发一次
  2. 懒加载时要监听计算滚动条的位置，但不必每次滑动都触发，可以降低计算的频率，而不必去浪费CPU资源

# 节流阀

类似于信号量

节流阀为空，可执行下次操作，否则不执行下次操作

# 防抖debounce

当事件被触发后，延迟n秒后再执行回调，如果在着n秒内事件又被触发，则重新计时**==（防抖能保证只有一次触发生效，前面的N多次触发都会被忽略）==**

# 应用场景

输入框防抖，用户在输入框输入内容时，可以通过防抖策略，只有输入完后才执行查询的请求，这样可以有效减少请求次数，节约请求资源
